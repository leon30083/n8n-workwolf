# Vibe Coding 指南

**通过与 AI 结对编程，将想法变为现实的终极工作方法论**

## 核心理念

**Vibe Coding** 是一种以**规划驱动**和**模块化**为核心的 AI 结对编程方法论。核心理念：

> **规划就是一切。** 谨慎让 AI 自主规划，否则你的代码库会变成一团无法管理的乱麻。

**三要素**：
1. **规划驱动** - 先结构，后代码，一定要规划好框架
2. **上下文固定** - 文档即上下文，不是事后补
3. **AI 结对执行** - 凡是 AI 能做的，就不要人工做

## 核心方法论

### 道·法·术三层体系

#### 道（核心原则）

```
- 凡是 AI 能做的，就不要人工做
- 一切问题问 AI
- 目的主导：一切动作围绕"目的"展开
- 上下文是第一性要素，垃圾进，垃圾出
- 先结构，后代码，一定要规划好框架
- 奥卡姆剃刀定理：如无必要，勿增代码
- 帕累托法则：关注重要的那20%
```

#### 法（工作方法）

```
- 一句话目标 + 非目标
- 能抄不写，不重复造轮子（Glue Coding 范式）
- 一定要看官方文档，先让 AI 下载到本地
- 按职责拆模块
- 接口先行，实现后补
- 一次只改一个模块
- 文档即上下文，不是事后补
```

#### 术（具体技巧）

```
- 明确写清：能改什么，不能改什么
- Debug 只给：预期 vs 实际 + 最小复现
- 测试可交给 AI，断言人审
- 代码一多就切会话
- AI 犯的错误整理为经验持久化存储
```

### 胶水编程（Glue Coding）

**软件工程的圣杯与银弹**

| 问题 | 解法 |
|:---|:---|
| AI 幻觉 | 只使用已验证的成熟代码，零幻觉 |
| 复杂性爆炸 | 每个模块都是久经考验的轮子 |
| 门槛过高 | 你只需要描述"连接方式" |

**核心理念**：能抄不写，能连不造，能复用不原创。

### Canvas 白板驱动开发

**图形化 AI 协作的新范式**

传统开发：代码 → 口头沟通 → 脑补架构 → 代码失控

Canvas 方式：**代码 ⇄ 白板 ⇄ AI ⇄ 人类**

| 痛点 | 解法 |
|:---|:---|
| AI 看不懂项目结构 | AI 直接读白板 JSON，秒懂架构 |
| 人类记不住复杂依赖 | 连线清晰，牵一发动全身 |
| 团队协作靠嘴说 | 指着白板讲，新人 5 分钟看懂 |

## Vibe Coding 工作流程

### 标准开发流程

```
1. 需求澄清
   - 一句话目标 + 非目标
   - 用黄金圈法则：是什么？为什么？怎么做？

2. 上下文文档
   - 创建 game-design-document.md 或 PRD
   - 创建 tech-stack.md（技术栈）
   - 创建 implementation-plan.md（实施计划）

3. 记忆库初始化
   - memory-bank/
     ├── game-design-document.md
     ├── tech-stack.md
     ├── implementation-plan.md
     ├── progress.md（进度记录）
     └── architecture.md（架构文档）

4. 规划执行
   - 永远先用 "Ask" 或 "Plan Mode" 确认
   - 每完成一步验证，再继续下一步
   - 每个里程碑更新 progress.md 和 architecture.md

5. 增量迭代
   - 完成基础版本
   - 添加细节功能（每个功能一个 feature-implementation.md）
```

### 提示词模式

#### 初始提示词模板

```
你是一个专业的 AI 编程助手。我想用 Vibe Coding 的方式开发一个项目。

请先问我：
1. 你想做什么项目？（一句话描述）
2. 你熟悉什么编程语言？（不熟悉也没关系）
3. 你的操作系统是什么？

然后帮我：
1. 推荐最简单的技术栈
2. 生成项目结构
3. 一步步指导我完成开发

要求：每完成一步问我是否成功，再继续下一步。
```

#### 实施提示词模板

```
1. 打开 progress.md 记录你做了什么
2. 把新的架构洞察添加到 architecture.md
```

## 编程思维要点

### 语言层要素（12 层框架）

1. **值层** - 基本类型、字面量
2. **引用层** - 变量、常量、绑定
3. **表达式层** - 运算、组合
4. **语句层** - 控制、顺序
5. **函数层** - 抽象、复用
6. **模块层** - 组织、依赖
7. **类型层** - 约束、检查
8. **异步层** - 并发、流
9. **错误层** - 异常、恢复
10. **元编程层** - 代码生成
11. **并发层** - 同步、通信
12. **系统层** - 外部交互

### 强前置条件约束

```
1. 必须有明确的目标描述
2. 必须先规划后编码
3. 必须模块化，禁止单体巨文件
4. 必须有验证测试
5. 必须记录进度和架构
```

### 常见坑汇总

| 坑点 | 解决方案 |
|:---|:---|
| AI 幻觉 | 用已验证代码，Glue Coding |
| 复杂性爆炸 | 按职责拆模块 |
| 上下文不足 | 文档先行，不是事后补 |
| 代码一团乱 | 先结构后代码 |
| AI 自主规划 | 谨慎让 AI 规划 |
